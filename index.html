软件需求分析复习指北

第一章 需求工程导论

软件项目的三种类型（P3）

  • 在预期的时间内，在预算的成本下完成预期的所有功能，则项目为成功项目
  • 已经完成，软件产品能够正常工作，但在生产中或者超支，或者超期，或者实现的功能不全的项目为问题项目
  • 因无法进行而被中途撤销，或者最终产品无法提交试用，则项目为失败项目

项目成功项目的因素（P4）

  • 用户参与
  • 高层管理支持
  • 清晰的需求说明
  • 正确的项目计划
  • 切合实际的期望
  • 细化的项目里程碑
  • 员工能力
  • 主人翁意识
  • 清晰的目标和前景
  • 努力工作

影响问题项目的因素（P4）

  • 缺少用户的输入
  • 不完整的需求说明
  • 需求变化
  • 缺乏高层管理支持
  • 技术能力不足
  • 缺乏资源
  • 不切实际的期望
  • 目标不清晰
  • 不现实的时间要求
  • 新技术的影响

影响失败项目的因素（P4）

  • 不完整的需求说明
  • 缺少用户输入
  • 缺乏资源
  • 不切实际的期望
  • 缺乏高层管理支持
  • 需求变化
  • 缺乏计划
  • 额外的无用功能
  • 缺乏IT管理
  • 技术能力不足

软件模拟特性（P5）

​ 导致需求问题的原因中最重要的原因就是：未能很好的理解和掌握“应用”型软件的模拟特性以及由此而产生的一系列影响和要求
​ 软件的模拟特性来源于其知识载体的特性：软件在运行中表现出来的特性，行为应该和应用的现实情况保持一致。这样，人们通过观察软件的表现就可以得出相应现实问题的答案，即软件“模拟”了现实

“模拟”性具体指

  • 目的性
  • 正确性
  • 现实可理解性

第二章 需求基础

问题解决的基础——模拟共享现象（P26）

处于问题域之外的解系统之所以能够解决问题域中问题，是因为问题域与解系统之间存在有效的互动，并在互动中相互影响。而问题域与解系统能够形成互动的基础是解系统部分模拟了问题域，这种模拟性称为共享现象

需求和问题的层次（P28）

  • 业务需求，针对整个业务的期望
  • 用户需求，针对具体任务的期望
  • 系统级需求，针对用户与系统一次交互的期望

优秀需求的特性（P43）

  • 完备性

    需求是否描述了开发人员设计和实现这项功能所需的所有信息

  • 正确性

    每项需取样都必须正确描述所需要的系统功能，要真实反应用户的意图

  • 可行性

    需求必须能够在系统及其运行环境的已知条件和约束下实现

  • 必要性

    每一项需求都应该是必要的，它是满足用户的业务需求说必需的

  • 无歧义

    需求要能够正确传递的前提是传递者和受众能够形成共同的理解

  • 可验证

    需求是可以通过分析、检查、模拟或测试等方法能够判断需取样是否被满足

给Express开启HTTPS

Posted on 2020-11-27 Edited on 2020-12-12 In Nodejs

因为最近有一个项目放在express上跑，而且还必须要开启https，其实我一开始是先通过有ssl的nginx来反向代理没有证书的express，但是，经过我多方面的尝试，最终以失败告终，所以，我决定狠下心来，来给express配置安装ssl证书

环境

  • CentOS 版本： 7.7
  • express 版本：4.17.1
  • node 版本： 14.15.1

步骤

  • 首先找到入口文件（一般是app.js），用vim打开
  • 配置以下信息

// 导入依赖的模块
const express = require(‘express’)
const fs = require(‘fs’)
const https = require(‘https’)
// 找到ssl证书和秘钥
const privateCrt = fs.readFileSync(<证书路径>)
const privateKey = fs.readFileSync(<秘钥路径>)
// https的配置信息
const HTTPS_OPTOIN = {
key: privateKey,
cert: privateCrt
};
/*
* 你的代码逻辑
*/
// 生成服务器
const httpsServer = https.createServer(HTTPS_OPTOIN, app);
httpsServer.listen(<你所定义的端口号>);

  • 启动服务

关于Mac上的MRT

Posted on 2020-10-24 Edited on 2020-12-12 In uncategorized

最近手贱升级了一下系统，然后上安卓课的时候发现MacBook的电池续航雪崩，原来我的Mac可以一直坚挺四节课，但是自从升级了之后，就只能卑微的使用两节课了，然后我就非常狼狈的回去拿了我的充电器，所以我狠下决心来，一定要解决这个非常恶心的bug。

原因

这个问题的原因的是Mac上的MRT无限的崩溃导致CPU疯狂被吃

这是我的系统崩溃的报告，下面全是MRT，简直就是离谱。。。

解决方案

我本来天真的以为，只要杀掉这个进程就可以解决了，显然我还是太年轻了。这个进程根本杀不掉，即使在终端把他kill掉之后，他还是会立马重新创建一个进程。
于是，我在国内外网各种收集资料，但是网上的所有方法都无效，就连给出的MRT的路径都在我的Mac上找不到！
但是在我绝望之际，我终于，找到了MRT的正确的路径

/Library/Apple/System/Library/CoreServices/MRT.app

你可以看到，这个MRT.app的版本是1.68的，这就是整个问题的所在，因为在这个版本，MRT就会无限的卡死，于是我找了我一个同学的Mac，然后看了他的版本，嗯，非常好，是1.67，于是，我就想把它的放到我的电脑上来，做一个降级的操作（文件会在最后面给出），但是，我的权限居然不够（我是root）！！
这是因为apple的怕root瞎操作，给一些特定的文件限制了连root都无法操作，但是，我还是找到了解决方案！

  1. 重启
  2. 听到“咚”的一声按下Command+R，进入恢复模式
  3. 打开终端输入csrutil disable

这时候，你就获得最高权限啦！！

然后在将老版本的MRT放入我上面给出的路径进行替换，美滋滋儿，问题解决了！MRT没有了！感动！

遗憾

这个基本上解决了问题，但是，在你重新启动电脑的时候，电脑还会自动安装MRT，所以还是会出现原来的问题，所以，我们坐等apple解决这个bug吧！

Basic Quality of Developers_02

Posted on 2020-10-09 Edited on 2020-12-12 In 软工职业素养

Research and Development Process

  • Project approval
  • Project plan
  • Requirement development
  • Design & Achievement
  • Test
  • Release
  • Node term

Proposal

This stage mainly includes project management and project review.

Project management mainly includes business demand research, market research, technical feasibility study, etc. In this process, Business Demand Research Report, Market Research Report, Technical Feasibility Analysis Report and Project Application Report shall be formed.

Project review shall organize a formal review of the Project Application Report.

The overall plan of the project includes determining the project scope, life cycle model,work breakdown, estimating project size, workload, schedule, cost, and required resources.

Subordinate plans mainly include configuration management plans, quality assurance plans, measurement and analysis plans, risk management plans, and training plans.

Test phase When the testing phase is completed, a formal review meeting is held to analyze the test report and results and make a decision on whether the product can be finalized

Release stage This stage mainly includes activities such as product confirmation (such as product trial operation, customer acceptance testing, etc.), product conversion (for hardware + embedded software products), and official product launch.

Closing phase This stage mainly includes archiving and confirmation of results, writing project summary reports, enriching project experience and process documents into the organization’s process wealth library, and enriching reusable resources generated in the project into the reusable basic library.

Professional Quality

  • A document habitGood documentation is an important part of the formal r&d process. As a developer, it is normal to spend 30% of your working time writing technical documentation. As a senior developer and systems analyst, the percentage is much higher. Without documentation, a software system will lack vitality, and will encounter great trouble in future error-checking, upgrade and module reuse.
  • Standardized and standardized coding habitsAs a formal software company’s rules, the variable naming of the code, the format of the comment in the code, and even the length of the indentation of the lines in the nest and the number of blank lines between functions are clearly defined. Good writing habits not only help code migration Error correction also facilitates collaboration between different technicians. The code has good readability and is a basic quality for developers.
  • Reusable, modular thinkingReusable design, modular thinking is that when developers complete any functional module or function, they must think more, don’t limit themselves to the simple idea of completing the current task, think about whether the module can be separated from this system Whether it can be directly referenced in other systems and application environments by simply modifying the parameters, so that repetitive development work can be greatly avoided.
  • Testing HabitsIn a formal software company, a full-time test engineer is indispensable, but it does not mean that with a full-time test engineer, developers can not perform self-tests; software development as a project, a very important feature is problem detection. The sooner it is solved, the lower the cost of solution. Developers carefully test each piece of code and each sub-module after completion, and can try to find and solve some potential problems as early as possible, so that the efficiency of the overall system construction and Maximum reliability is guaranteed.

Personal Quality

  • Requirements understanding ability Developers need to understand the full requirements of a module. Many developers write programs that focus on a single functional requirement. They attribute performance metrics to the hardware, operating system, and development environment, while ignoring the performance considerations of their code.Among the performance requirements, stability, support capability and security are very important. As a developer, it is necessary to evaluate the environment in which the module is operating, the load pressure it will be subjected to, and the possibility of various potential dangers and malicious attacks.
  • Ability to learn and summarize Good at learning, for any career, is necessary to move forward, for developers, this kind of demand is even higher.But study also want to find the target, blind pursuit of some superficial, superficial things and nouns, will never have a qualitative improvement.Good at summary, is also a reflection of learning ability, each completion of a research and development task, complete a piece of code, should have a purpose to track the application of the program and user feedback, summary at any time, find their own shortcomings, so that gradually improve, a developer can grow up.

Career Development

  • The First Difficulty： threat from tiger Developers have to face a lot of hungry tigers. For example, leaders seem to be the natural enemies of developers. The most common complaints of developers are what they don’t know about technology, what they don’t get and what they don’t pay. Developers can only face the reality when they encounter such problems.
  • The Second Difficulity：Bill’s competition Generally speaking, most of the developers in China eat “youth meal”, and most of them are in the golden age of 24-28 years old. By the age of 30, a lot of young developers will bring you great competitive pressure. Learning ability, salary requirements, life burden, etc. will make you almost never have the courage to start from scratch. As the old Chinese saying goes, “the waves behind the Yangtze River are pushing the waves ahead, and a generation of new people are replacing the old ones”. This historical law is especially obvious and cruel in the software development industry. If a person has been working in personnel development, the inevitable result is to reduce wages and even unemployment.
  • The Third Difficulity：Self strength We all know that the most difficult thing for people to overcome is themselves, so self is the biggest obstacle that developers need to overcome. Most developers regard system analysts and project managers as their career goals, but the achievement of these goals requires personal quality, market opportunities and many other conditions. Too many developers just waste their time in the state of high failure and low failure
  • Coping with difficulties：Make career planning Quality of system analyst: customer demand analysis ability, system architecture and design ability, module decomposition design ability, project process control ability, project risk assessment ability, etc. Quality of Project Manager: team organization ability, communication and coordination ability, problem-solving ability and good professional ethics. Personal development can only rely on personal learning and efforts of developers.

Three periods and characteristics of computer development

  • The program design
  • Application system
  • Software engineering

software crisis

Three periods in the development of computer software

  • Early era (before the mid-1960s) the programming phaseGeneral hardware, special software; The program is small in size, and the writer and user are the same person (in the same group)
  • Second generation (mid-1960s - mid-1970s) program system phaseEmergence of “software workshop”, product software; A “personalized” development approach.
  • Third generation (after mid-1970s) software engineering phaseSoftware development has become a new engineering discipline - software engineering.

what is a software crisis?

Software crisis refers to a series of serious problems encountered in the development and maintenance of computer software.

two main problem

  • How to develop software, how to meet the growing demand for software.
  • How to maintain an ever-expanding supply of software

Major manifestations of the software crisis

  • Inaccurate estimates of software development costs and schedules.
  • The user is not satisfied
  • Software quality is not high, reliability is poor
  • Software is often unmaintainable and errors are difficult to correct.
  • Lack of proper documentation
  • The software cost accounts for the system total cost proportion to rise year by year
  • The speed of software development can not keep pace with that of computers

The cause of the software crisis

  • It’s about the nature of the software itself.
  • Software is not easy to maintain
  • In the process of software development, more or less adopted the wrong methods and techniques.
  • Without a complete and accurate understanding of the user’s needs, I rushed into writing the program.

How to eliminate the software crisis?

  • Technical measuresUse better software development methods and tools
  • The software development of organization and management This is not a mysterious skill of individual labor, but a kind of engineering project which is well organized, tightly managed, coordinated and completed by all kinds of personnel.

software engineering

What is software engineering?

Software engineering is an engineering subject that guides the development and maintenance of computer software. It adopts the concept, principle, technology and method of engineering to develop and maintain software, which combines the management technology which has been proved to be correct by time test with the best technical method that can be obtained at present. Software engineering is a subject involving the research and application of the principles, methods and tools of software planning, requirement analysis, design, coding, testing and maintenance.

Why is the software life cycle divided into phases?

One of the common strategies used by human beings in solving complex problems is to break down each problem, that is, to decompose the problem and then solve each sub problem separately. The “life cycle method” adopted in software engineering is to decompose the complex problems of software development and maintenance from the perspective of time. It divides the long life cycle of software into several stages in turn. Each stage has relatively independent tasks, and then gradually complete the tasks of each stage.

How to divide it?

Software life cycle is generally divided into three periods:

  • software definition (problem definition, feasibility study, requirement analysis)
  • software development (overall design, detailed design, coding and unit testing, comprehensive testing)
  • software maintenance

Software development process model

  • waterfall model**advantages**

    • Accepts the work object of the activity from the previous activity as input 。

    • What should be done to implement the activity using this input

    • Give the output of the activity to the next activity

    • Review the implementation of the activity.If its work is confirmed, it will proceed to the next activity.

    • *disadvantages**

    • From the perspective of epistemology, human cognition is a process that is repeated many times and cannot be completed at once. However, the several stages divided in the waterfall model do not reflect the repetitive nature of this cognitive process.

    • Software development is a knowledge-intensive development activity that requires mutual cooperation to complete, but the waterfall model does not reflect this.

  • Prototype model**advantages**

    • The prototype method is much better than the traditional life cycle method in getting a good definition of requirements. It can deal with fuzzy requirements, and developers and users can fully communicate.

    • The prototype system can be used as a training environment, which is conducive to the synchronization of user training and development, and the development process is also a learning process.

    • The prototype gives the user the opportunity to change the unreasonable final system in mind.

    • The prototype can develop a flexible computer system with low risk.

    • prototypes increase opportunities to make the system more maintainable and user-friendly.

    • The prototype reduces the total development cost and time.

    • *disadvantages**

    • Model effects or insight into areas that developers are not familiar with using the minor parts as the main framework to make irrelevant prototypes

    • Prototype iterations do not converge to the developer’s prior goal of making each change, and in order to eliminate errors, the minor parts get larger and larger, overwhelming the major parts

    • The prototype converges too quickly to the requirements set and ignores some basic points

    • Resource planning and management can be difficult, and updating documents at any time can be troublesome

    • Long-term development on the prototype environment, only pay attention to the satisfactory prototype, easy to forget the differences between the user environment and the prototype environment

  • Fountain modelThe fountain model assumes that the phases of the software life cycle overlap and iterateMainly used in object-oriented methods

  • Helix modelBased on the prototype, the spiral model is formed by repeated prototype iteration and increased risk assessment

prepare detailed design specification

  • determine the algorithm of each module, express the algorithm process with tools, and write out the detailed process description of the module.
  • determine the data structure of each module.
  • determine module interface details. Detailed design is the forerunner of coding.

The advantages of structural programming technology

  • The method of gradual refinement from top to bottom conforms to the general law of human solving complex problems, so it can significantly improve the success rate and productivity of software development projects.
  • The program developed by the process of first global and then local, first whole and then detail, abstract and then concrete step-by-step refinement has a clear hierarchy, so it is easy to read and understand.
  • Do not use GO TO statement only use a single entry single exit control structure, so that the static structure of the program and its dynamic implementation is more consistent, easy to read and understand.
  • The control structure has a definite logical pattern, the writing of program code is limited to a few straightforward ways, so the source program is clear and smooth.
  • The clarity and modularity of the program make it possible to reuse the most code when modifying and redesigning a software.
  • The logical structure of the program is clear and is conducive to the proof of program correctness.

Tools for Process Design

  • Program flowchart
  • Box diagram (N-S diagram)
    • Functional domain (that is, the scope of a specific control structure) is clear.
    • Impossible to transfer control arbitrarily.
    • Easily determine the scope of local and global data.
    • It is easy to express nested relationships, and it can also represent the hierarchical structure of modules
  • Problem Analysis Chart (PAD)

  • Determination tree and judging table
  • Process Design Language (PDL) (Pseudo code)
